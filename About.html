The need for yet another meta config language tool


I've used a ton of config languages in various tool and projects, and have universally been dissatisfied by the results.


I've tried TOML, which is mostly great, but resulted in ambiguities with the [[ ]] notation and was hard to support.

I've used YAML, but pretty much hated the strictness on spacing

I've tried JSON, but was quite disappointed with the lack of comments, and the rigidity around comma placement.

I looked at SANE, but didn't like the adamant belief that block quotes shouldn't be allowed (they're super useful for commenting out a section of code/config and trying something else in a pinch)

I looked ad SDLANG, but didn't like their forcing of numbers to 32/64 bit values.

I looked at dhall.  Lots of good stuff there.  No implementation in C or C++ yet, though, and I'm not sure I'm going to be the one to cover all that it can do...

So, those are the starting points for coming up with something of my own.  But... how do I come up with something of my own?

Goals for a config parser tool

I would like a tool that allows me to specify in a 'simple' way the format for input, and then I'd like that tool to help me as a programmer validate user input, and help the user to create valid input.

I'd like the same meta-description that is used to parse the input to be used to help the user.  I'd like man-page style help descriptions available for correcting config files and command line arguments.  I'd like the ability to add or create new grammars at run time so that things like modules can specify their own syntax and be added to the configuration system dynamically.

Support UTF-8 and alternative character sets.

Re-usability
- Same description
    parses the config file
    validates the config file
    generates documentation for the format of the config file
    provides user help for fixing/editing/creating the config file

- Same description language (and maybe even description)
    parses argv command line input
    provides documentation for the format of the argv input
    generates documentation for the format of the input file

- Same description language (and maybe even description)
    parses cli input
    provides input completions
    provides realtime help by knowing the required input format
    generates documentation for the cli

Reformat an input in a "canonical" order and format
Load a config file from one format and output to another format (translate)

I'd also like the back end to be agnostic from the front end.

In other words, you should be able to load a 'TOML' description and configure your program with a TOML config file, or load a JSON config file and configure that same program with a JSON config file.  If you want your special version of JSON in your config file to ... say ... SUPPORT COMMENTS -- like EVEN BLOCK COMMENTS ... you should be able to adjust the JSON input file and do that.

My goals are
    1. Make it easy on the USER.  Be gracious and try to understand what they want.
    2. Enable scientific configuration.  Focus on math, units, and ease of specifying things CORRECTLY
    3. Figure out how to HELP the user.  Make validation and errors CLEAR and helpful.
    4. Make it clean and easy to read (as much as can be figured out, anyway)
    5. Make it easy to comment out blocks to try new things
